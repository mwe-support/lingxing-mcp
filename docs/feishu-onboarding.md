## 这个 MCP 能干什么

`Lingxing MCP` 的目标不是替代领星 ERP，也不是替代 Amazon API。  
它做的是把领星 ERP 的读取能力，整理成一套 AI 工具可以直接调用的入口。

更适合这些场景：

- 多店铺统一查看销量、订单、促销
- 统一查看 ASIN 的浏览、会话、广告和销量
- 补广告周报、ASIN 周报、经营周报这类表格
- 下载并解析 Amazon 报告
- 做利润、库存、补货和源表排查
- 团队成员在不同网络、不同地点协作
- 想让 Claude Code、Codex、Cursor 直接读领星数据
- 不想把主凭证散落在每个人电脑里

它现在能做的事情，主要包括下面几类：

- 基础映射：店铺列表、站点、时区、广告账号
- 店铺经营：店铺销量、ASIN 日报、订单、促销
- 产品表现：浏览、会话、广告、销量等汇总指标
- 广告报表：SP / SD / SB 的 campaign、product ad、keyword、target、search term，以及小时级报表
- 广告管理：受控更新 SP 活动、广告组、关键词、投放、商品广告，以及新增 / 归档否定关键词和否定投放
- 利润分析：店铺、ASIN、父 ASIN 维度利润
- 亚马逊源表：订单、库存、交易、预留库存
- FBA / 补货：库存汇总、库存明细、补货建议和限制
- 报告导出：创建任务、查询任务、刷新链接、下载并解析文件
- 高层聚合：按 ASIN 汇总广告日数据、周数据
- 做最小健康检查和烟测

当前公开版优先 **只读**，广告管理工具默认 `dry_run=true` 且需要显式确认，因为团队真正的第一步通常是：

- 先把白名单跑通
- 先把固定出口跑通
- 先把 AI 接入跑通

而不是一上来就做写操作。

## 怎么安装这个 MCP

### 方式 1：个人本地验证

如果你只是自己先试一下，可以用本地 `stdio` 模式。

适合：

- 个人验证
- 单机测试
- 你自己持有领星主凭证

这不是团队共享的主方案。

### 方式 2：团队共享

如果你要给团队共享，推荐用 `HTTP gateway`。

适合：

- 团队多人一起用
- 白名单受限
- 大家会换网络、开 VPN、异地办公

推荐 IDE：

- Claude Code
- Codex
- Cursor

推荐做法是直接对 AI 说：

```text
请从 GitHub 仓库 `lingxing-mcp` 接入领星 MCP 到当前工作区。主路线请采用“固定出口云服务器 + HTTP gateway + Claude Code / Codex / Cursor”方案；如果当前环境只是在我个人电脑上做临时验证，再改用 stdio。请同时检查示例配置、环境变量、依赖和最小健康检查是否完整。
```

如果你要手动安装，看：

- [manual-install.md](./manual-install.md)

一句话总结安装路线：

- **主路线**：固定出口云服务器 + HTTP gateway
- **验证路线**：本地 `stdio`

## 人在领星后台要做什么

这一部分必须由 **领星超级管理员** 操作。

登录后，进入：

`设置 -> 业务配置 -> 基础 -> 开放接口`

你需要完成：

1. 添加固定出口服务器的 IP 白名单
2. 添加 API 账号
3. 记录 `APP ID`
4. 记录 `AppSecret`

建议流程：

- 如果你只是想先跑通，可以先用较宽的只读权限验证链路
- 跑通后，再把权限收缩到实际需要的范围

更完整的管理员清单看：

- [admin-setup.md](./admin-setup.md)

## 人在腾讯云和服务器上要做什么

### 腾讯云侧

你需要先买一台固定出口服务器。  
推荐默认值：

- 腾讯云 `CVM`
- `2核4G`
- `Ubuntu 22.04 LTS`
- 基础带宽即可
- 长期共享建议绑定 `EIP`

如果你从没买过云服务器，直接看：

- [tencent-cloud-buying-guide.md](./tencent-cloud-buying-guide.md)

### 服务器侧

买完后，通常要做这几件事：

1. 登录服务器
2. 配置 `LINGXING_APP_ID`
3. 配置 `LINGXING_APP_SECRET`
4. 初始化 token / tokens 文件
5. 启动 `HTTP gateway`
6. 做 `healthz`、`lingxing_health_check`
7. 再做 `lingxing_smoke_check`
8. 如需验证扩展能力，再补一条 `lingxing_product_performance` 或广告报表调用

### 客户端侧

团队成员通常只需要：

1. 拿到网关地址
2. 拿到自己的成员令牌
3. 在 Claude Code / Codex / Cursor 里接入 MCP 配置
4. 先做一次最小健康检查
5. 再按业务场景验证：
   - 经营数据：`lingxing_store_sales` / `lingxing_order_lists`
   - 流量广告：`lingxing_product_performance` / `lingxing_ads_sp_*`
   - 周汇总：`lingxing_asin_weekly_rollup`

## 常见问题

### 为什么不能直接每个人都本地直连

因为领星 OpenAPI 有 IP 白名单限制。  
多人、多地点、多 VPN 的情况下，本地直连会让白名单很快失控。

### 为什么推荐固定出口

固定出口能让领星只看到一个稳定 IP。  
这样团队成员换网络时，不需要反复改白名单。

### 为什么当前先做只读优先

因为只读是最低风险、最容易先跑通访问链路的方式。  
只有在读取链路稳定后，再考虑写接口，整体风险才可控。当前已有的广告管理工具必须先 dry run 查看请求体，只有传入 `confirm=true` 且 `dry_run=false` 才会真正调用领星写接口。

### 它是不是 Amazon API 的替代品

不完全是，但对很多卖家团队来说，它已经能覆盖很大一部分“店铺经营 API + 广告 API”的查询场景，并支持少量受控 SP 广告管理动作。
特别是当团队更关心的是销量、订单、广告、流量、利润、库存、补货和报告下载，而不是直接操作 Amazon 官方原始接口时，`Lingxing MCP` 会更贴近实际经营分析流程。

### 为什么同事不能直接拿主密钥

因为 `APP ID / AppSecret` 应该只保留在服务器。  
同事应该使用自己的成员令牌，而不是直接拿主密钥。

## 安全提醒

> [!WARNING]
> 下面这些值不要发群、不要截图外传、不要提交到 Git：
>
> - 真实 `APP ID`
> - 真实 `AppSecret`
> - 真实 Bearer Token
> - 真实 `tokens.json`
> - 真实服务器 IP
> - 团队内部接入地址

截图传播前，至少要打码：

- `APP ID`
- `AppSecret`
- 白名单 IP
- 服务器地址
- Bearer / token

如果怀疑泄露，先做这几件事：

1. 轮换 API 账号或重置 `AppSecret`
2. 轮换成员令牌
3. 更新服务端配置
4. 再继续分发接入信息
