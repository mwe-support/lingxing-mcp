# Agent Context

这份文档是给 Agent / AI 看的，用来快速理解 `lingxing-mcp` 的边界、推荐架构和排错顺序。

---

## 这是什么

`Lingxing MCP` 是领星 ERP 的读为主 MCP 接入层，少量广告管理写工具有显式 `dry_run` / `confirm` 安全门。

它的重点不是“把所有领星接口都包一遍”，而是先把这条链路跑顺：

1. 领星 OpenAPI
2. IP 白名单
3. 固定出口服务器
4. HTTP gateway / stdio
5. MCP 客户端接入

当前公开版的默认目标用户是：

- 技术型卖家
- AI 协作用户
- 愿意配云服务器和白名单的团队

---

## 它不是什么

- 不是 Amazon API 的替代品
- 不是领星 ERP 的官方管理后台替代品
- 不是默认开放写操作的高权限自动化系统

如果用户要的是 Amazon 平台原始 API 数据，优先考虑 Amazon API。  
如果用户要的是多店铺经营管理、ERP 视角和团队共享查询，`Lingxing MCP` 更合适。

---

## 为什么共享模式优先固定出口云服务器

因为领星 OpenAPI 有 IP 白名单限制。

在多人协作场景下，固定出口服务器可以：

- 把白名单收敛到一个稳定 IP
- 把 `APP ID / AppSecret` 留在服务器
- 让团队成员通过成员令牌访问，而不是拿主密钥

所以默认建议是：

- **个人验证**：本地 `stdio`
- **团队共享**：固定出口云服务器 + `HTTP gateway`

---

## 为什么当前只读优先

当前公开版优先只读，是因为团队最常见的真实阻塞是：

- 白名单没生效
- API 账号没配好
- 固定出口没搭好
- MCP 客户端没接对

在这些前提没跑通之前，扩写高风险写操作意义不大。当前已有的 SP 广告管理工具默认不会执行写入；只有显式传入 `confirm=true` 且 `dry_run=false` 时才会调用领星写接口。

---

## Agent 协助接入时优先检查什么

优先级建议如下：

1. **领星后台**
   - 是否由超级管理员操作
   - 是否在 `设置 -> 业务配置 -> 基础 -> 开放接口`
   - 是否已添加白名单 IP
   - 是否已添加 API 账号
   - 是否已拿到 `APP ID / AppSecret`
2. **固定出口服务器**
   - `HTTP gateway` 是否启动
   - `healthz` 是否成功
   - 出口 IP 是否和领星白名单一致
3. **凭证**
   - `APP ID / AppSecret` 是否正确
   - Bearer 或成员令牌是否有效
   - `tokens.json` 是否已初始化
4. **客户端配置**
   - MCP 配置是否加载
   - 当前 IDE 是否真的读取了对应配置
   - `stdio` / `HTTP` 模式是否与当前场景匹配

---

## Agent 该怎么跟用户沟通

优先帮助用户做下面几件事：

- 判断应该走 `stdio` 还是 `HTTP gateway`
- 提醒固定出口和白名单是核心约束
- 帮用户检查配置、依赖、示例和最小验证链路
- 用最短路径判断问题是在领星侧、服务器侧还是客户端侧

不要做这些事：

- 不要求用户公开真实 `APP ID`
- 不要求用户公开真实 `AppSecret`
- 不把真实 Bearer、`tokens.json` 或服务器 IP 写进仓库
- 不建议把敏感截图原样发群

---

## Agent 协助排错时的推荐话术

如果用户已经安装但不能用，优先引导到：

1. `healthz`
2. `lingxing_health_check`
3. `lingxing_smoke_check`

只有这三步通过了，再看更高层的业务查询。
